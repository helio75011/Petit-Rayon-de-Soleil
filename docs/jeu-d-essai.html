Restauration / réinitialisation

En cas d'incident (base corrompue, données de test polluées par des essais manuels) ou pour faire repartir les tests de zéro, un script dédié remet l'application dans un état propre et connu — en une seule commande, avec vérification automatique.

Restaurer en une commande : npm run db:reset

Le script scripts/reset-db.js enchaîne les quatre étapes exigées et refuse de s'exécuter si une base de production Turso est configurée (garde-fou anti-effacement). Voici sa sortie réelle :

bash — npm run db:reset
$ npm run db:reset

[1/4]  Suppression de la base locale... OK (1 fichier(s) retiré(s))
[2/4]  Création des tables (init)... OK
[3/4]  Réinjection des données de démonstration (seed)... OK
[4/4]  Vérification :
       ✓ admin@soleil.fr  (admin)
       ✓ user@soleil.fr  (user)
       ✓ camille@soleil.fr  (user)
       ✓ messages validés  : 5 (attendu 5)
       ✓ messages en attente : 2 (attendu 2)
       ✓ messages rejetés  : 1 (attendu 1)
       ✓ favoris           : 2 (attendu 2)

Base restaurée avec succès. Tous les comptes et volumes sont conformes.

Les quatre étapes du script

  1. Suppression de la base locale — efface soleil.db et ses fichiers annexes (-wal, -shm).
  2. Relance de l'initialisation — recrée les tables via init().
  3. Réinjection des données de démonstration — insère le jeu d'essai via seed().
  4. Vérification des comptes de test — contrôle que les 3 comptes attendus sont présents avec le bon rôle, et que les volumes (5 validés, 2 en attente, 1 rejeté, 2 favoris) sont exacts. Le script sort en erreur (exit 1) si un seul contrôle échoue.
Note

Fermer d'abord le serveur (npm start) : sous Windows, un serveur en cours verrouille soleil.db et empêche sa suppression.

Méthode manuelle équivalente

Sans le script, les mêmes étapes se font à la main :

bash — restauration manuelle
# 1. Supprimer la base actuelle (et ses fichiers annexes WAL/SHM)
rm -f soleil.db soleil.db-wal soleil.db-shm

# 2. Recréer les tables + réinsérer le jeu d'essai complet
npm run init-db

Comment fonctionne la restauration

La commande npm run init-db exécute node db.js --init, qui appelle init() (création des tables via CREATE TABLE IF NOT EXISTS) puis seed() (insertion du jeu d'essai). La fonction seed() est idempotente : si des comptes existent déjà, elle ne réinsère rien — d'où l'étape 1 qui repart d'une base vide.

db.js — garde-fou d'idempotence
async function seed() {
  const count = await db.prepare('SELECT COUNT(*) AS n FROM users').get();
  if (count && count.n > 0) return;   // déjà peuplée → on ne duplique pas
  // … insertion des comptes, messages et favoris …
}

Cas de la base en ligne (Turso)

La même commande initialise la base de production, en la préfixant des variables d'environnement Turso :

bash — initialisation de la base Turso
TURSO_DATABASE_URL=libsql://<base>.turso.io \
TURSO_AUTH_TOKEN=<jeton> \
  npm run init-db
