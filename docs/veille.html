Veille sécurité & bonnes pratiques

La sécurité du déploiement s'appuie sur un référentiel reconnu (OWASP) et sur des pratiques déjà appliquées dans le code. Chaque carte indique la pratique et son état actuel dans le projet.

Secrets Sécurité des secrets

Les identifiants sensibles (jeton Turso, URL de base) ne doivent jamais se retrouver dans le dépôt. Ils sont fournis uniquement par variables d'environnement, et le fichier .env est exclu du versionnement.

État projet : appliqué — .env et .vercel listés dans .gitignore ; aucun secret en dur dans le code (vérifié). Veille : rotation des jetons et bonnes pratiques de gestion des secrets Vercel.

OWASP Bonnes pratiques OWASP

Le Top 10 OWASP sert de grille de lecture des risques web courants (injection, mauvaise authentification, contrôle d'accès défaillant…). C'est la référence suivie pour la sécurité applicative.

État projet : plusieurs risques déjà couverts — injection SQL neutralisée par requêtes paramétrées ; mots de passe hachés (bcrypt) ; contrôle d'accès via middlewares requireAuth/requireAdmin. Veille : évolutions du Top 10 et nouvelles recommandations.

Packages Mises à jour des packages

Une dépendance obsolète est une porte d'entrée potentielle. La politique est double : peu de dépendances (surface réduite) et mise à jour régulière des versions, priorisée sur les correctifs de sécurité.

État projet : 3 dépendances runtime seulement. Contrôle par npm audit — 0 vulnérabilité à ce jour. Veille : suivi des correctifs via Dependabot.

Contrôle de sécurité des dépendances

La commande ci-dessous, exécutable à tout moment et intégrable au pipeline CI, dresse l'état des vulnérabilités connues :

bash — npm audit
$ npm audit
found 0 vulnerabilities

De la veille à l'action

La veille n'a de valeur que si elle déclenche une réaction. Le circuit est le suivant :

  • Alerte reçue (Dependabot, npm audit, changelog) → évaluation de la criticité.
  • Correctif de sécurité → mise à jour de la dépendance, puis npm test (15 tests) pour vérifier l'absence de régression.
  • Validation par la CI → le pipeline GitHub Actions rejoue installation, qualité et tests avant toute fusion.
  • Déploiement → une fois au vert, Vercel remet en ligne automatiquement.
En résumé. La veille (Dependabot, npm audit, changelogs Node/Vercel/Turso) alimente un cycle court : détecter → corriger → tester → déployer. Les dépendances minimales et les pratiques OWASP déjà en place rendent ce cycle rapide et peu risqué.
