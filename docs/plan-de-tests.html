01

Stratégie de test

La stratégie repose sur des tests d'intégration automatisés exécutés de bout en bout contre le serveur réel : chaque test démarre l'application Express, envoie de vraies requêtes HTTP via fetch, et vérifie le code de statut et le corps de la réponse. Cette approche valide simultanément le routage, la logique métier, l'accès aux données SQLite et la sécurité — au plus près de l'usage réel.

Le choix a été fait de couvrir en priorité les exigences Must du MoSCoW et les exigences non fonctionnelles de sécurité, plutôt que de viser une couverture exhaustive de chaque route :

  • Cas nominal — un message aléatoire est bien renvoyé (page d'accueil).
  • Authentification — inscription puis connexion valides ; connexion refusée avec un mauvais mot de passe.
  • Contrôle d'accès — une route protégée refuse l'anonyme (401) ; une route admin refuse un utilisateur normal (403).
  • Parcours métier complet — proposition d'un message → invisible tant qu'il n'est pas approuvé → approbation admin → visible.
  • Conformité RGPD — la suppression de compte efface l'utilisateur, qui ne peut plus se connecter.
Outil retenu : le module natif node:test (assertions node:assert), sans aucune dépendance de test à installer. Cohérent avec la philosophie minimale du projet et exécutable partout où Node est présent.
02

Environnement de test

Les tests s'exécutent en local, sur une base de données isolée. Le hook before() supprime l'éventuel fichier soleil.db puis le recrée et le remplit (seed) avant la première assertion ; le serveur écoute sur un port dédié 3999 pour ne pas entrer en conflit avec une instance de développement.

Procédure d'exécution

  1. Installer les dépendances applicatives : npm install.
  2. S'assurer qu'aucune instance du serveur ne tient soleil.db ouverte (sinon le verrou de fichier empêche la réinitialisation de la base sous Windows).
  3. Lancer la suite : npm test (raccourci de node --test).

Jeu d'essai détaillé — fonctionnalité la plus représentative

La fonctionnalité retenue est le parcours proposition → approbation (test #6). Elle traverse toutes les couches de l'application et implique les deux rôles (utilisateur et administrateur), ce qui en fait le scénario le plus démonstratif du bon fonctionnement du système.

Décomposition étape par étape : entrée → attendu → obtenu → écart.
ÉtapeEntréeAttenduObtenuÉcart
1 · Proposer Utilisateur de démo user@soleil.fr connecté envoie POST /api/messages avec { content: "Tu es capable de grandes choses." } 201, message créé avec approved = 0 201 —
2 · Vérifier l'invisibilité GET /api/messages (liste publique) le message n'apparaît pas absent —
3 · Approuver Admin connecté envoie POST /api/admin/approve/:id 200, passage à approved = 1 200 —
4 · Vérifier la visibilité GET /api/messages à nouveau le message apparaît désormais présent —
Aucun écart constaté. Le comportement observé correspond exactement au comportement attendu à chaque étape. Le cycle de modération fonctionne : un message reste privé tant qu'un administrateur ne l'a pas approuvé.

Piste d'amélioration

Pour renforcer le volet sécurité du plan de tests, un test explicite d'anti-injection SQL pourrait être ajouté (envoi d'une charge du type ' OR '1'='1 dans un champ, et vérification qu'elle est traitée comme une donnée littérale grâce aux requêtes paramétrées, sans effet sur la base).
