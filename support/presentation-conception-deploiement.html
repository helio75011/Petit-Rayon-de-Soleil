03 UML — cas d'utilisation

Le use case répond à : « qui peut faire quoi ? »

  • Les acteurs externes sont le visiteur, l'utilisateur inscrit et l'administrateur.
  • Les ovales représentent les services rendus par le système, non les écrans.
  • Le diagramme vérifie la couverture des besoins et les droits par rôle.
  • Le cas significatif retenu est : proposition puis approbation d'un message.
Ne pas confondre

Le cas d'utilisation décrit le besoin fonctionnel, pas les tables, les routes ou les classes.

04 UML — diagramme de séquence

La séquence répond à : « dans quel ordre les composants échangent-ils ? »

sequenceDiagram

    participant F as Front
    participant E as Express
    participant A as requireAuth
    participant D as Base
    actor Ad as Admin

    M->>F: Rédige un message
    F->>E: POST /api/messages (Bearer token)
    E->>A: Vérifie la session
    A->>D: Contrôle token et utilisateur
    D-->>A: Utilisateur validé
    A-->>E: Identité reconstituée
    E->>D: INSERT status = 'pending'
    D-->>E: Message créé (201)
    E-->>F: Statut « en attente »

    Ad->>E: POST /api/admin/approve/:id
    E->>D: Vérifie le rôle admin
    D-->>E: Rôle confirmé
    E->>D: UPDATE status = 'approved'
    D-->>E: Message publié (200)
    E-->>Ad: Confirmation

Le message est inséré avec status = 'pending'. Après contrôle du rôle par requireAdmin, la base passe status à 'approved'.

06 Merise — le MCD

Il décrit les entités métier, les associations et les cardinalités, sans parler de tables ni de SQL.

erDiagram
    UTILISATEUR ||--o{ MESSAGE : "propose (1,N)"
    UTILISATEUR ||--o{ SESSION : "possede (1,N)"
    UTILISATEUR }o--o{ MESSAGE : "met en favori (N,N)"

    UTILISATEUR {
        string nom
        string email
        string role
    }
    MESSAGE {
        string contenu
        string statut
        date date_creation
    }
    SESSION {
        string jeton
        date date_creation
    }
  • UTILISATEUR propose des MESSAGE — relation de 1 à N.
  • UTILISATEUR possède des SESSION — relation de 1 à N.
  • Un utilisateur met plusieurs messages en favori et inversement — relation N,N.
  • Le MCD traduit le vocabulaire du métier avant tout choix de stockage.

07 Merise — le MLD

Il convertit le MCD en structure relationnelle : tables, clés primaires et clés étrangères.

erDiagram
    USERS ||--o{ MESSAGES : propose
    USERS ||--o{ SESSIONS : authentifie
    USERS ||--o{ FAVORIS : ajoute
    MESSAGES ||--o{ FAVORIS : "figure dans"

    USERS {
        int id PK
        string name
        string email UK
        string password_hash
        string role
        date created_at
    }
    MESSAGES {
        int id PK
        string content
        string status "pending|approved|rejected"
        int propose_par FK
        date created_at
    }
    FAVORIS {
        int user_id PK_FK
        int message_id PK_FK
    }
    SESSIONS {
        string token PK
        int user_id FK
        date created_at
    }
Transformation MCD → MLD

Les entités deviennent users, messages et sessions. La relation 1,N devient une clé étrangère — propose_par dans messages. La relation N,N « favori » devient une table d'association favoris à clé primaire composite. Le MLD reste indépendant du moteur de base de données.

08 Merise — le MPD

L'implémentation cible pour SQLite : types, contraintes, valeurs par défaut et comportements de suppression.

  • users.email est unique et obligatoire.
  • password_hash est stocké à la place du mot de passe en clair.
  • messages.status vaut 'pending' par défaut, donc en attente de validation, avec une contrainte CHECK limitant les valeurs à pending, approved et rejected.
  • favoris possède une clé primaire composée (user_id, message_id).
  • Les clés étrangères encadrent les suppressions : CASCADE sur favoris et sessions, SET NULL sur propose_par.
Pourquoi un statut et pas un booléen

Un booléen ne code que deux états : validé ou non. La règle métier en distingue trois, car il faut conserver la trace d'une décision de rejet. Le CHECK fait respecter cette règle par la base elle-même, pas seulement par le code applicatif.

11 Qualité et stratégie de tests

Le parcours proposition puis approbation traverse toutes les couches et les deux rôles.

  • Unitaire — fonctions isolées : hachage des mots de passe, extraction du token.
  • Intégration — route, logique et base reliées par de vraies requêtes HTTP.
  • Système et acceptation — parcours complet et vérification manuelle.
  • Qualité — ESLint, Prettier et node:test rejoués automatiquement dans la CI.

12 CI/CD — les définitions à maîtriser

La CI vérifie, le CD livre ou déploie. Le vocabulaire doit être précis.

CI Intégration continue

  • Déclenchée à chaque push ou pull request.
  • Récupère le code, installe, contrôle la qualité, exécute les tests.
  • Objectif : détecter rapidement une régression.

CD Continuous delivery

  • Le livrable est prêt à être déployé après validation.
  • Une approbation humaine peut rester nécessaire.
  • Objectif : rendre la livraison fiable et répétable.

CD Continuous deployment

  • Chaque modification validée est publiée automatiquement.
  • Vercel redéploie à partir du dépôt Git.
  • Objectif : automatiser aussi la mise en ligne.
Réponse courte

« GitHub Actions est le gardien de la qualité. Vercel est le mécanisme de publication. »

13 Le pipeline réel

À chaque push ou pull request sur main, le code est vérifié avant toute suite.

2Node.jsInstallation du runtime, version 20
3Installnpm ci
4Qualitélint puis format:check
5Testsnpm test — node:test

Vert : la qualité et les tests ont réussi. Rouge : lire le journal, corriger le test ou la règle, puis repousser le code.

14 Déploiement — Vercel et Turso

Vercel est serverless : un fichier SQLite local ne garantit pas une persistance fiable en production.

→NavigateurHTTPS
→VercelFonction serverless
→ExpressRoutes et métier
→TursoSQLite hébergé via libSQL
Sécurité et exploitation

Les secrets TURSO_DATABASE_URL et TURSO_AUTH_TOKEN sont définis dans les variables d'environnement, jamais dans le code ni dans Git. Vercel est connecté au dépôt GitHub et redéploie après un push sur main.

15 Ce qui est automatisé, ce qui reste manuel

Auto Automatisé

  • GitHub Actions : installation, qualité et tests à chaque push ou PR.
  • Vercel : build et publication depuis la branche principale.
  • predeploy : tests relancés avant un déploiement CLI.
  • Scripts npm : setup, init-db, test, deploy.

Manuel À renforcer

  • Relecture humaine avant fusion de la PR.
  • Migrations et restauration de base.
  • Tests visuels de bout en bout dans un navigateur.
  • Garde stricte de déploiement uniquement après CI verte, à finaliser.
À expliquer

« Je n'automatise pas aveuglément. J'automatise les répétitions vérifiables et je conserve une décision humaine pour les choix sensibles. »
