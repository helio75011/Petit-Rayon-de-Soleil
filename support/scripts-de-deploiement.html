Scripts de déploiement

Commandes exactes, reproductibles et documentées, du premier clonage du projet jusqu'au redéploiement après correction. Toutes les commandes sont enregistrées comme scripts npm dans package.json.

Projet Petit Rayon de Soleil Auteur de Breyne Hélio Runtime Node.js 18+ Cible Vercel + Turso
Où sont définis ces scripts ? Dans le bloc "scripts" de package.json. On les lance avec npm run <nom> (sauf npm start et npm test qui n'ont pas besoin de run). Les documenter ici garantit qu'un autre développeur — ou le jury — peut rejouer chaque étape à l'identique.
1

Initialisation du projet

Récupérer le code source sur la machine de développement.

bash
# Cloner le dépôt puis entrer dans le dossier
git clone https://github.com/<votre-compte>/petit-rayon-de-soleil.git
cd petit-rayon-de-soleil

Prérequis : Git et Node.js 18 ou supérieur installés (node --version pour vérifier).

2

Installation

Installer les dépendances déclarées dans package.json (Express, bcryptjs, client libSQL).

bash
npm install

Crée le dossier node_modules/. Aucune configuration supplémentaire n'est nécessaire à cette étape.

3

Initialisation de la base de données

Créer les tables et insérer les données de départ (un admin, un utilisateur de démo, cinq messages).

bash
npm run init-db
→ Base initialisée avec un admin, un utilisateur et 5 messages.
→   Admin : admin@soleil.fr / admin123
→   User  : user@soleil.fr  / user1234

En local : crée le fichier soleil.db. Pour la base en ligne (Turso) : préfixer avec les variables d'environnement (voir §5).

Raccourci : npm run setup enchaîne l'installation et l'initialisation de la base (npm install && npm run init-db) — pratique pour préparer un environnement neuf en une seule commande.
4

Tests

Vérifier que les parcours critiques fonctionnent avant tout déploiement (module natif node:test).

bash
npm test
→ tests 7
→ pass  7
→ fail  0

Couvre l'inscription, la connexion, la sécurité des routes (401 / 403), le parcours proposition → approbation admin, et la suppression de compte (RGPD).

Note

Fermer le serveur (npm start) avant de lancer les tests : ils recréent une base propre et un serveur en cours peut verrouiller le fichier soleil.db.

5

Déploiement

Mettre l'application en ligne sur Vercel. Deux voies possibles.

Voie A — via l'interface Vercel (recommandée, la première fois)

Après avoir poussé le projet sur GitHub, importer le dépôt sur vercel.com, renseigner les deux variables d'environnement, puis cliquer sur Deploy.

Variables d'environnement Vercel
TURSO_DATABASE_URL = libsql://<votre-base>.turso.io
TURSO_AUTH_TOKEN   = <jeton fourni par Turso>
Sécurité

Le jeton TURSO_AUTH_TOKEN ne doit jamais être écrit dans le code ni commité. Il est fourni uniquement via les variables d'environnement (le fichier .env est exclu par .gitignore).

Voie B — en ligne de commande (Vercel CLI)

Une fois le projet lié à Vercel, un déploiement de production se lance en une commande :

bash
# Une seule fois : installer l'outil et lier le projet
npm install -g vercel
vercel link

# Déployer en production (joue d'abord les tests via le hook predeploy)
npm run deploy

npm run deploy exécute vercel --prod. Grâce au hook predeploy, npm lance automatiquement npm test juste avant : si un test échoue, le déploiement est annulé.

6

Redéploiement après correction

Remettre en ligne une version corrigée du code.

bash
# 1. Corriger le code, puis vérifier localement
npm test

# 2. Publier la correction
git add .
git commit -m "fix: correction du bug X"
npm run redeploy      # = git push

Redéploiement automatique : Vercel est connecté au dépôt GitHub. Chaque git push sur la branche principale déclenche un nouveau build et une mise en ligne, sans action manuelle. La commande npm run redeploy n'est donc qu'un alias explicite de git push.

Alternative CLI : pour forcer un redéploiement sans passer par Git, on peut relancer npm run deploy (vercel --prod) directement.
