Priorisation des fonctionnalités de l'application « Petit Rayon de Soleil ». Chaque exigence est reliée à son point d'entrée réel dans le code et à son état d'avancement.

Implémenté et testé Non implémenté (piste) Hors périmètre (assumé)

Must have

Indispensable le produit n'a pas de sens sans

  • ✔ Inscription & connexionPOST /api/register · POST /api/login
  • ✔ Afficher les messages bienveillantsGET /api/messages
  • ✔ Proposer un messagePOST /api/messages (status = 'pending')
  • ✔ Modérer : approuver / rejeterPOST /api/admin/approve/:id · DELETE /api/admin/reject/:id

Should have

Important attendu, mais contournable

  • ✔ Mettre un message en favoriGET / POST / DELETE /api/favoris
  • ✔ Gérer ses informations personnellesGET /api/me · PUT /api/me
  • ✔ Supprimer son compte (RGPD)DELETE /api/me · suppression en cascade

Could have

Souhaitable valeur ajoutée, si le temps le permet

  • ✔ Message aléatoire (« message du jour »)GET /api/messages/random
  • • Recherche / filtre des messagesnon implémenté — axe d'évolution

Won't have (this time)

Écarté hors périmètre de cette version

  • — Notifications (e-mail / push)reporté à une version ultérieure
  • — Rôles avancés / permissions finesdeux rôles suffisent : user & admin
4Must — tous livrés

Lecture de la matrice. La totalité des exigences Must et Should est implémentée et couverte par les tests automatisés : le périmètre engageant du projet est entièrement tenu. Côté Could, le message aléatoire est livré ; la recherche/filtre constitue un axe d'évolution clairement identifié. Les éléments Won't sont des choix de cadrage assumés pour cette version.
