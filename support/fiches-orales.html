Fiches orales — conception, Merise, UML et CI/CD

Phrase d'ouverture

« Je présente le projet en suivant la logique complète du développement : analyser le besoin, modéliser les interactions et les données, implémenter, tester puis déployer. »

L'ordre de conception

La question « par lequel on commence ? » est quasi certaine. Cette séquence est la réponse.

  1. Besoins et acteurs — qui fait quoi, règles métier, contraintes.
  2. Cas d'utilisation UML — les fonctionnalités par rôle.
  3. Diagramme de séquence UML — le scénario critique dans le temps.
  4. MCD, puis MLD, puis MPD — les données, du métier au SGBD.
  5. Architecture en couches, implémentation, tests, CI/CD, production.
Réponse courte

« Je commence par les besoins et les acteurs. Je modélise les interactions avec UML. Ensuite je modélise les données avec Merise, du MCD au MPD, avant d'implémenter la base et de tester le scénario le plus important. »

Merise et UML — ne pas les confondre

Ce sont deux méthodes distinctes, complémentaires : elles ne répondent pas à la même question.

Merise

Décrit les données. Produit le MCD, le MLD et le MPD.

UML

Décrit le comportement. Produit les cas d'utilisation et le diagramme de séquence.

Formule à retenir : Merise structure ce que le système mémorise, UML décrit ce qu'il fait.

Les trois niveaux Merise

MCD Conceptuel

Vision métier : entités, associations, cardinalités. Aucune table, aucun SQL.

MLD Logique

Traduction relationnelle : tables, clés primaires, clés étrangères, table de liaison. Indépendant du moteur.

MPD Physique

Implémentation SQLite : types, contraintes, valeurs par défaut, règles de suppression.

Phrase à retenir

« Le MCD explique le métier, le MLD structure le relationnel, le MPD configure la base réellement exécutée. »

Les règles de passage

C'est ce que le jury attend derrière « savoir l'expliquer ».

MCD → MLD
Chaque entité devient une table, son identifiant devient la clé primaire. Une association 1,N devient une clé étrangère du côté N : « propose » place propose_par dans messages. Une association N,N ne peut pas se réduire à une clé étrangère : elle crée une table de liaison — c'est l'origine de favoris, dont la clé primaire est le couple (user_id, message_id).
MLD → MPD
On choisit le SGBD et on descend dans le concret : types SQLite (INTEGER, TEXT), valeurs par défaut, contraintes de domaine (CHECK) et règles de suppression. C'est ici que se décide le comportement RGPD : CASCADE sur favoris et sessions, SET NULL sur propose_par pour anonymiser sans perdre les messages publiés.

UML — les deux diagrammes du dossier

1. Cas d'utilisation
Répond à « qui peut faire quoi ? ». Trois acteurs : visiteur, utilisateur inscrit, administrateur. Les ovales sont les services rendus, pas les écrans. Sert à vérifier la couverture des besoins et les droits par rôle.
2. Séquence
Répond à « dans quel ordre les composants échangent-ils ? ». Détaille le scénario proposition puis approbation : le front envoie POST /api/messages avec un Bearer token, Express appelle requireAuth, la base vérifie session et utilisateur, le message est inséré avec status = 'pending'. L'admin appelle ensuite POST /api/admin/approve/:id, requireAdmin vérifie le rôle, puis la base passe status à 'approved'.

Questions probables

Q. Pourquoi commencer par le MCD ?
Parce qu'il exprime le besoin métier sans subir les choix techniques. Je définis d'abord les entités, les associations et les cardinalités, puis je transforme en tables. Si je commençais par le SQL, je figerais des décisions techniques avant d'avoir compris le métier.
Q. Pourquoi une table favoris ?
La relation entre utilisateurs et messages est N,N : un utilisateur met plusieurs messages en favori, un message peut être favori de plusieurs utilisateurs. Une clé étrangère ne suffit pas, il faut une table d'association portant les deux clés. Sa clé primaire composite garantit qu'on ne peut pas enregistrer deux fois le même favori.
Q. Pourquoi status = 'pending' au départ ?
C'est la règle métier de modération : le message existe en base mais reste invisible au public jusqu'à la décision de l'administrateur. J'ai choisi un statut à trois valeurs plutôt qu'un booléen, parce qu'un booléen ne saurait pas distinguer « en attente » de « rejeté » : il me fallait garder la trace d'une décision négative. La contrainte CHECK (status IN ('pending','approved','rejected')) fait respecter la règle par la base elle-même, pas seulement par le code.
Q. Différence entre cas d'utilisation et séquence ?
Le cas d'utilisation donne la vue fonctionnelle : acteurs et services. La séquence détaille le déroulement temporel d'un scénario précis entre le front, l'API, l'authentification et la base.
Q. Que se passe-t-il à la suppression d'un compte ?
Trois comportements distincts, décidés au niveau du MPD. Les sessions et les favoris sont supprimés en CASCADE. Les messages publiés sont conservés mais anonymisés : la clé étrangère propose_par passe à NULL grâce à ON DELETE SET NULL. C'est un arbitrage RGPD assumé : on efface la donnée personnelle sans détruire le contenu déjà validé et visible publiquement.
Q. Pourquoi Turso plutôt que SQLite local sur Vercel ?
Vercel est serverless et son système de fichiers est éphémère : un fichier SQLite local n'y garantit aucune persistance. Turso fournit une base SQLite hébergée, accessible par le réseau via libSQL, ce qui conserve la logique SQL du projet tout en étant compatible avec ce type d'hébergement.
Q. CI et CD, quelle différence ?
La CI vérifie automatiquement le code à chaque push ou pull request. Le CD prépare ou effectue la livraison. Formule courte : « GitHub Actions est le gardien de la qualité, Vercel est le mécanisme de publication. » Distinguer aussi continuous delivery (livrable prêt, approbation humaine possible) et continuous deployment (publication automatique).
Q. Que fait précisément le pipeline ?
Cinq étapes à chaque push ou PR sur main : checkout du code, installation de Node.js, npm ci, contrôles de qualité (npm run lint puis npm run format:check), enfin npm test. Une erreur rend le job rouge et signale la fusion en échec.
Q. Qu'est-ce qui reste manuel ?
La relecture humaine avant fusion, les tests visuels de bout en bout, les migrations et la restauration de base, et la décision de pousser une version sur main. À dire tel quel : « Je n'automatise pas aveuglément. J'automatise les répétitions vérifiables et je conserve une décision humaine pour les choix sensibles. »

Ne pas survendre

Dire ce qui est automatisé aujourd'hui et ce qui reste une évolution — notamment le verrouillage strict du déploiement après CI verte, qui n'est pas encore en place. Distinguer clairement local, test et production. Répondre par le raisonnement et la preuve, jamais par une technologie seule.
