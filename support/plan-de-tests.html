01

Stratégie et niveaux de test

Le plan couvre les quatre niveaux de test attendus, du plus fin (une fonction isolée) au plus global (le parcours utilisateur complet). Les niveaux unitaire et intégration sont automatisés (14 tests, module natif node:test) ; les niveaux système et acceptation sont validés par ces tests de bout en bout complétés d'une vérification manuelle des parcours.

02

Environnement de test

Les tests s'exécutent en local. Les tests unitaires n'ont besoin d'aucune base. Les tests d'intégration travaillent sur une base réinitialisée à neuf : le hook before() supprime le fichier soleil.db puis le recrée et le remplit (seed) avant la première assertion ; le serveur écoute sur un port dédié 3999 pour ne pas entrer en conflit avec une instance de développement.

Système
Windows 10 (win32 x64)
Runtime
Node.js 18+
Base de données
SQLite / libSQL via @libsql/client
Port de test
3999 (isolé du port applicatif 3000)
Framework de test
node:test + node:assert (natif)
Jeu de données
seed : 1 admin, 1 utilisateur, 5 messages

Procédure d'exécution

  1. Installer les dépendances applicatives : npm install.
  2. S'assurer qu'aucune instance du serveur ne tient soleil.db ouverte (sinon le verrou de fichier empêche la réinitialisation de la base sous Windows).
  3. Lancer la suite complète : npm test (raccourci de node --test) — les deux fichiers de tests sont découverts automatiquement.

04

Tests d'intégration, système & acceptation tests/api.test.js

Ces tests démarrent l'application Express réelle et envoient de vraies requêtes HTTP via fetch. Ils valident donc à la fois l'intégration des couches (route → métier → base), le fonctionnement système de l'application entière, et l'acceptation des exigences Must du cahier des charges. Jeu d'essai détaillé — fonctionnalité la plus représentative

La fonctionnalité retenue est le parcours proposition → approbation (test système #6). Elle traverse toutes les couches de l'application et implique les deux rôles (utilisateur et administrateur), ce qui en fait le scénario le plus démonstratif du bon fonctionnement du système.

Décomposition étape par étape : entrée → attendu → obtenu → écart.
ÉtapeEntréeAttenduObtenuÉcart
1 · Proposer Utilisateur de démo user@soleil.fr connecté envoie POST /api/messages avec { content: "Tu es capable de grandes choses." } 201, message créé avec status = 'pending' 201 —
2 · Vérifier l'invisibilité GET /api/messages (liste publique) le message n'apparaît pas absent —
3 · Approuver Admin connecté envoie POST /api/admin/approve/:id 200, passage à status = 'approved' 200 —
4 · Vérifier la visibilité GET /api/messages à nouveau le message apparaît désormais présent —
Aucun écart constaté. Le comportement observé correspond exactement au comportement attendu à chaque étape. Le cycle de modération fonctionne : un message reste privé tant qu'un administrateur ne l'a pas approuvé.
