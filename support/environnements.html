Environnements : développement, test, production

Les trois environnements de Petit Rayon de Soleil exécutent le même code. Seule la configuration change, pilotée par des variables d'environnement — jamais par une modification du code source.

2

Rôle de chaque environnement

Local (développement). C'est l'environnement de travail quotidien. La base est un simple fichier soleil.db créé par npm run init-db : aucun service externe, aucune connexion réseau, tout est sur la machine. On y développe et on essaie les fonctionnalités avant de les valider.

Test. Lancé par npm test, il sert à vérifier automatiquement les parcours critiques (inscription, connexion, sécurité 401/403, proposition → approbation, RGPD). Avant chaque campagne, la base est supprimée puis recréée à neuf et remplie de données d'essai : chaque exécution part donc d'un état propre et reproductible, indépendant de ce qu'on a pu bricoler en local. Le serveur de test écoute sur le port 3999, distinct du port 3000, pour ne pas entrer en conflit.

Production. C'est l'application déployée sur Vercel, accessible au public en HTTPS. La base n'est plus un fichier local mais Turso (SQLite hébergé, compatible serverless), indispensable car le système de fichiers de Vercel est éphémère. Les données y sont réelles et persistantes.

Comment s'opère la bascule

Un seul point de décision, dans db.js : si l'URL Turso est présente, on se connecte à la base hébergée ; sinon, on retombe sur le fichier local.

db.js — sélection de la base
const client = createClient(
  process.env.TURSO_DATABASE_URL          // définie → production
    ? { url: process.env.TURSO_DATABASE_URL,
        authToken: process.env.TURSO_AUTH_TOKEN }
    : { url: 'file:soleil.db' }          // absente → local / test
);

Le même mécanisme vaut pour le port : process.env.PORT || 3000. En local il vaut 3000, en test le fichier de test le force à 3999, et en production Vercel l'injecte automatiquement.

Pourquoi un seul code ? Faire tourner les trois environnements sur la même base de code garantit que ce qui est testé est exactement ce qui est déployé. On élimine toute une classe de bugs du type « ça marchait en local mais pas en prod », puisque seule la configuration diffère, pas la logique.
Sécurité

Les variables de production (TURSO_AUTH_TOKEN notamment) ne sont jamais stockées dans le code ni commitées : elles sont renseignées dans l'interface Vercel, et le fichier .env local est exclu par .gitignore.
